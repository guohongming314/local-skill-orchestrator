# ADR: Codex host adapter for hard tool routing

- Status: Accepted
- Date: 2026-07-15
- Issue: #130
- Parent epic: #118
- Evidence baseline: `codex-cli 0.144.1`

## Context

The project Skill and Context Capsule implement soft routing: they tell Codex which
capabilities belong to the current phase, but they cannot remove tools that the host already
loaded. Hard routing must make the capsule selection authoritative without making the existing
soft-routing path depend on a host-specific adapter.

The design requires dynamic capsule recompilation at phase boundaries and explicitly separates
confirmed state from irrelevant intermediate context. The roadmap defers hard routing to a Codex
Host Adapter and MCP Gateway. This spike evaluates the concrete Codex surface available for that
layer.

## Decision drivers

1. A thread must not see an unselected MCP tool.
2. Different concurrent threads must be able to receive different selections.
3. A phase transition must have a deterministic exposure boundary.
4. The mechanism must not edit the user's durable Codex configuration as ordinary operation.
5. Disabling or not installing the adapter must preserve today's soft-routing behavior.
6. The contract must leave room for tool-level filtering, not only whole-server filtering.

## Evidence and proof-of-concept notes

All experiments used two local JSONL MCP servers. `alpha` advertised `alpha_tool`; `beta`
advertised `beta_tool`. Visibility was inspected through the generated app-server protocol's
`mcpServerStatus/list` request with a `threadId`.

### 1. Per-run generated Codex configuration/profile

`codex --help` in 0.144.1 exposes both dotted `--config key=value` overrides and
`--profile <CONFIG_PROFILE_V2>`. A generated profile was layered over a base config containing
both servers, with one server set to `enabled = false` per phase. In this installation,
`codex -p phase-a mcp list --json` still reported both inherited servers enabled. A profile that
only names selected servers therefore does not provide deny-by-omission, and the observed profile
layer did not provide a reliable narrowing boundary.

A completely isolated generated `CODEX_HOME/config.toml` can provide startup-only isolation, but
it also changes the home used for authentication, thread persistence, caches, and user settings.
Copying or linking those assets would create secret-handling and lifecycle complexity. This is a
useful diagnostic fallback, not the primary adapter.

**PoC result:** configuration can determine exposure at process/thread startup, but generated
profiles are not accepted as the hard-routing boundary. Do not mutate the user's base config.

### 2. MCP gateway proxy

A one-server PoC configured Codex only with a local `gateway` MCP server. The gateway owned two
candidate tool definitions but returned only `alpha_tool` from `tools/list` for the selected
phase. `mcpServerStatus/list` showed one server and exactly that tool; `beta_tool` was absent.
Calls to an unselected tool were rejected by the gateway.

This proves a gateway can enforce tool-level allowlisting and can hide individual tools from a
server that otherwise exposes a larger catalog. It also creates the correct location for policy,
audit, upstream connection management, and fail-closed call validation.

A gateway alone does not prove that Codex will rediscover a changed `tools/list` during an active
turn. Codex caches the discovered MCP inventory. A host-controlled reload or a new phase session
is still required at the transition boundary.

**PoC result:** selected tool definitions can be exposed and unselected calls can be denied. Use
the gateway when a capability selection is finer than a whole configured MCP server.

### 3. App-server-native thread configuration and reload

The schema generated by `codex app-server generate-json-schema` contains:

- `thread/start.config`: an arbitrary thread configuration object;
- `thread/resume.config`: configuration overrides for a resumed thread;
- `mcpServerStatus/list`, optionally scoped by `threadId`;
- `config/mcpServer/reload`;
- `config/batchWrite.reloadUserConfig`, documented to hot-reload user config into all loaded
  threads.

A PoC started two concurrent threads against the same app-server process:

- thread A received `config.mcp_servers.alpha.enabled=true` and
  `config.mcp_servers.beta.enabled=false`;
- thread B received the inverse configuration.

The status request for thread A contained only `alpha_tool`; thread B contained only `beta_tool`.
This is a working per-thread, deny-by-explicit-disable boundary without editing durable user
configuration.

A second PoC rewrote the base config between phases and invoked `config/mcpServer/reload`.
The same loaded thread changed from `alpha_tool` to `beta_tool`. This demonstrates between-turn
mutability, but the reload API is process-wide/user-config-oriented rather than a thread-scoped
configuration update. It can therefore perturb unrelated loaded threads and is not safe as the
normal multi-session phase-transition mechanism.

`turn/start` has no MCP configuration field. No protocol method in the generated 0.144.1 schema
updates MCP exposure for exactly one already-loaded thread. Tool exposure must not be changed in
the middle of an active turn.

**PoC result:** app-server thread configuration is the primary whole-server control at
`thread/start`/`thread/resume`. Process-wide reload is suitable for diagnostics or a dedicated
single-thread app-server only, not shared production routing.

## Decision

Adopt a **Codex Host Adapter with app-server-native per-thread configuration**, and compose it with
an **MCP Gateway for tool-level filtering**.

1. At a phase boundary, the control plane compiles a fresh capsule and calls the adapter before it
   starts or resumes Codex work for that phase.
2. The adapter passes explicit `enabled = true/false` overrides for every known MCP server in
   `thread/start.config` or `thread/resume.config`. Deny-by-omission is not sufficient.
3. When the capsule selects only part of an upstream server, Codex sees one project gateway server;
   the gateway exposes only the selected tools and rejects unselected calls even if a stale client
   attempts one.
4. Exposure is immutable during a turn. A phase transition waits for turn completion, checkpoints
   confirmed state, and establishes the next phase session boundary before sending new work.
5. Until a thread-scoped MCP reconfiguration method exists and is proven, the safe default is a
   **phase session handoff** (start a new thread with the freshly compiled confirmed-state capsule).
   A dedicated one-thread app-server may instead reload and resume, but a shared app-server must
   not use process-wide config reload for routine transitions.
6. The user's durable Codex config is never rewritten by normal routing.

The phase handoff changes the Codex thread ID and does not preserve unconfirmed conversational
context. That is an intentional user-visible tradeoff: progress must be surfaced as “starting
phase N with X tools,” and only confirmed state crosses the boundary. It matches the design's
recompile-not-append rule, but callers must not promise one uninterrupted host thread across
phases.

## HostAdapter contract

The portable contract remains:

```python
expose(capabilities, phase) -> session_config
```

For the Codex adapter, `session_config` has the following semantic fields (the implementation may
use typed models with equivalent names):

```text
phase                 stable phase identifier
selected_capabilities ordered capability IDs from the compiled capsule
thread_config         complete per-thread Codex config overrides, including explicit MCP enables
mcp_gateway           absent, or gateway endpoint plus exact allowed server/tool IDs
transition            start | resume | handoff
exposure_digest       digest of phase + selected capabilities + effective routing config
```

Required invariants:

- the adapter consumes validated capsule output; it does not make capability-selection decisions;
- every known MCP server is explicitly enabled or disabled in `thread_config`;
- the gateway allowlist is exact and fail-closed;
- the exposure digest is recorded with the run/checkpoint and invalidates on phase or capability
  changes;
- configuration is prepared before the first turn of the phase;
- an unsupported host returns an unsupported result rather than pretending hard routing occurred.

The runner treats “no adapter configured” and “adapter unsupported” as **soft-routing mode**:
compile and send the same Context Capsule used today, start Codex through the existing path, emit a
clear diagnostic that hard isolation is unavailable, and never reject an otherwise valid task
solely because hard routing is absent. Policy may separately require hard routing for high-risk
work; that is not the default contract.

## Mid-session mutability answer

- **During an active turn:** unsupported and prohibited.
- **Between turns in one dedicated app-server process:** technically possible through MCP config
  reload, proven by the PoC, but process-wide and therefore unsafe for shared concurrent threads.
- **Per loaded thread:** startup/resume configuration is proven; an in-place thread-scoped MCP
  update is not present in the 0.144.1 generated protocol.
- **Recommended phase transition:** establish a new phase session boundary with fresh per-thread
  config (normally a thread handoff), preserving only confirmed state.

## Re-scoping follow-up issues

- **E15.2:** build the gateway as a fail-closed tool-level allowlist and call validator; do not make
  it responsible for workflow or capability selection.
- **E15.3:** implement `HostAdapter.expose`, explicit server disables, exposure digests, and phase
  handoff. Do not depend on global config reload. Rename any acceptance language promising the same
  Codex thread across phases to “same task run with confirmed-state continuity.”
- **E15.4:** run sandbox preflight before a newly installed capability can enter either the
  per-thread enable set or gateway allowlist.
- **E15.5:** assert visibility through `mcpServerStatus/list` and an attempted denied call; cover two
  concurrent threads with disjoint selections and the adapter-disabled soft-routing path.

## Consequences

Hard routing becomes evidence-based and deterministic at phase boundaries. Whole-server isolation
uses a native Codex surface, while the gateway supplies the missing tool-level boundary. Concurrent
threads can have disjoint configurations without user-config mutation.

The cost is explicit phase-session lifecycle management and possible thread handoff. Tool changes
are not instantaneous, and unconfirmed chat context does not cross a handoff. The adapter also
needs compatibility checks because app-server is experimental; if required fields or methods are
missing, it degrades to soft routing with a visible diagnostic rather than silently claiming
isolation.
