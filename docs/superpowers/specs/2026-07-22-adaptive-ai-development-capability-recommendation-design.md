# 自适应 AI 开发能力推荐设计

**日期：** 2026-07-22

**状态：** 对话方向已确认，等待书面规格复核

**产品入口：** Codex 原生 Bootstrap Skill 与确定性的 `vibe` 能力工具

## 问题

当前初始化能够盘点部分本地能力、推导实践包需求并生成项目配置，但推荐体验仍然偏向静态规则，且存在关键状态丢失：

- `permissions.network`、项目写入和命令执行虽然出现在访谈问题中，却没有正式进入 Blueprint 权限模型；除目标、生命周期和风险外，访谈回答主要依赖模型自由解释。
- “用户未回答”“尚未批准本次只读发现”“项目禁止网络”“候选运行时需要网络”没有被建模为不同状态，外层对话可能把未授权发现误述为用户禁用了网络。
- `quality.gates` 和 `repository.exploration` 被明确配置为没有默认候选，因此初始化会报告缺口，却不能给出下一步推荐。
- Superpowers 一类方案设计工作流、代码优化与审查、代码索引、跨会话记忆等能力没有形成完整、统一的需求域。
- CodeGraph 等候选使用单一仓库规模规则过滤，无法综合语言、模块耦合、现有索引、变更频率和用户偏好判断价值。
- 存在远程缺口时，外层流程可能直接进入项目配置 ChangeSet 审批，没有先完成候选发现、比较和用户选择。

产品需要从“按项目类型输出固定工具清单”转向“基于证据识别能力需求，再动态寻找和解释具体提供者”。用户举出的 Ponytail、Superpowers、Claude-Mem、CodeGraph、Chrome MCP 等名称是应当被搜索和比较的候选，不是所有项目必须安装的固定套餐。

## 已确认的产品原则

采用“先分析并提出建议，再询问少量关键偏好”的交互方式：

1. 先读取仓库证据和本地 Codex 能力，不要求用户重复可推断信息。
2. 先形成一版有理由的能力建议。
3. 只在答案会改变能力需求、候选排序、权限边界或安装范围时追问用户。
4. 问题逐个提出，并说明该问题会影响什么决策。
5. 未回答不能被解释为拒绝或禁止。

## 目标

1. 为开发闭环识别抽象能力需求，而不是套用固定产品模板。
2. 将仓库证据、用户偏好、本地能力、远程证据和组织策略组合成可解释的候选排名。
3. 可靠区分权限、发现授权和候选运行需求，消除虚假的“用户禁用网络”。
4. 对每个重要缺口给出有行动价值的本地线索或远程候选，不再输出无解释的裸缺口。
5. 在候选选择完成前禁止进入最终安装 ChangeSet 审批。
6. 保留本地优先、最小权限、不可变版本、事务写入、审计和 Doctor 验证。

## 非目标

- 不保证任何具体第三方产品出现在每个项目的最终推荐中。
- 不把模型自由生成的产品名称直接当作可安装证据。
- 不因 GitHub Star、下载量或流行度自动批准高权限候选。
- 不在用户未批准时执行网络发现、下载候选制品或安装能力。
- 不用新的推荐层替代 Codex 在日常任务中的原生 Skill 发现。

## 开发闭环能力模型

初始化首先推导能力域，然后为能力域解析提供者。首批开发闭环至少覆盖：

| 阶段 | 抽象能力示例 | 候选方向示例 |
| --- | --- | --- |
| 理解 | 仓库探索、符号检索、代码关系与影响分析 | 原生搜索、语言服务器、CodeGraph 类索引 |
| 设计 | 需求澄清、方案比较、实施计划、风险评审 | Superpowers 类工作流、项目设计 Skill |
| 实现 | 代码生成、重构、代码优化、项目规范执行 | 代码优化 Skill、语言/框架专用 Skill、CLI 或 Plugin |
| 验证 | 单元、集成、数据库、API、浏览器与可访问性测试 | 项目原生测试器、Playwright、Chrome DevTools MCP |
| 质量 | 格式化、静态检查、类型检查、安全扫描、质量门禁 | 仓库已有脚本、语言工具链、质量门禁 Skill |
| 连续性 | 跨会话项目记忆、决策记录、上下文恢复 | 项目日志、Claude-Mem 类能力 |
| 恢复 | Git 恢复、迁移回滚、部署回滚 | Git CLI、项目部署与数据库工具 |

候选方向仅用于帮助搜索和解释。解析器必须允许新提供者进入竞争，不能通过产品名称白名单决定最终结果。

## 证据模型

每个能力需求保存以下证据：

- 仓库事实：语言、框架、规模、monorepo、模块数量、测试和部署配置、数据库、浏览器界面、现有脚本。
- 工作事实：项目目标、生命周期、风险等级、交付约束和团队协作方式。
- 本地能力：已安装 Skill、Plugin、MCP、CLI、Hook、项目脚本及其验证状态。
- 用户偏好：明确回答、接受的系统建议和被锁定的决定。
- 远程证据：来源、发布者、不可变版本、权限、维护、采用度、静态扫描和兼容性。
- 策略证据：组织允许的来源、权限上限、数据处理和安装范围。

任何“推荐”“不推荐”或“无需安装”的结论都必须引用至少一项证据。缺少证据时应显示未知并提出针对性问题，而不是补全用户意图。

## 权限与授权状态

Blueprint 增加正式的权限和授权模型，每项记录值、来源、确认状态和适用范围。

### 项目操作权限

- `read_project`
- `write_project`
- `execute_command`
- `write_outside_project`
- `access_secrets`

### 网络相关状态

- `network_policy = unknown | denied | allowed-readonly | allowed`
- `discovery_approval = not-requested | approved | denied | expired`
- `artifact_fetch_approval = not-requested | approved | denied | expired`
- `candidate_runtime_network = none | readonly | readwrite | unknown`

这些状态不得互相推导。例如：

- 初始化访谈线程自身禁止联网，不表示项目网络策略为 `denied`。
- `discovery_approval = not-requested` 不表示用户拒绝网络。
- 用户批准只读候选搜索，不表示批准安装或候选运行时联网。
- 候选需要网络是候选属性，不是用户偏好。

所有权限回答都必须经过确定性字段映射并写入来源信息。模型可以总结自然语言，但不能成为权限状态的唯一存储层。

## 自适应推荐流程

```text
仓库与本地环境检查
→ 推导开发阶段和抽象能力需求
→ 解析已经满足的本地能力
→ 生成第一版建议、缺口和不确定项
→ 只询问会改变决策的关键偏好
→ 重新计算需求和本地候选排名
→ 若关键缺口值得远程发现，单独请求只读搜索授权
→ 搜索、验证、过滤并排序远程候选
→ 用户接受、替换、拒绝或延后候选
→ 生成精确权限和文件变更预览
→ 单独请求安装 ChangeSet 批准
→ 应用、Doctor 验证和结果说明
```

### 动态提问规则

问题只有满足以下至少一个条件时才提出：

- 不同答案会启用或禁用一个能力域；
- 不同答案会改变默认前三名候选；
- 不同答案会改变权限等级或数据边界；
- 不同答案会使安装范围从项目级扩大；
- 仓库证据相互冲突且无法安全推断。

问题应附带当前建议和影响，例如：

> 当前项目已有 Playwright 配置，优先建议复用它做可重复的浏览器测试。你是否还需要交互式浏览器调试？如果需要，我会继续比较 Chrome DevTools MCP 类候选。

不应询问能够从仓库可靠得出的语言、框架、测试命令或部署平台。

## 候选生成与搜索

候选来自三层：

1. 已验证本地提供者，包括项目脚本和 Codex 用户级能力。
2. 静态搜索线索，依据能力语义、生态类别和已知产品别名构造查询，但明确标为未验证。
3. 经用户授权后从 MCP Registry、skills.sh、GitHub 和组织目录获得的远程候选。

搜索查询不能只使用内部能力 ID。例如 `code.optimization` 应扩展为与项目语言和目标相关的查询词，如代码优化、重构、性能分析、代码审查以及用户提及的产品名。用户提到的新产品名作为高优先级搜索线索，但仍需验证其真实身份、能力和维护状态。

若用户输入的名称可能拼写错误或存在多个同名产品，系统应展示识别不确定性并在远程发现中解析，而不是静默绑定到某个预设产品。

## 排序与解释

候选先经过权限、来源、不可变身份、兼容性和安全硬过滤，再计算可解释分数：

| 维度 | 建议权重 | 说明 |
| --- | ---: | --- |
| 项目适配 | 30 | 是否匹配语言、框架、仓库规模、任务阶段和现有工具链 |
| 能力覆盖 | 20 | 覆盖多少已确认缺口，是否减少重复工具 |
| 可信与安全 | 20 | 来源、发布者、不可变身份、扫描结果和权限最小化 |
| 集成成本 | 10 | 安装、配置、常驻服务、索引、CI 和维护负担 |
| 维护状态 | 10 | 最近更新、发布节奏、归档状态和问题响应 |
| 用户偏好 | 5 | 用户明确偏好、拒绝和接受的工作方式 |
| 社区采用 | 5 | 仅作次要信号，不能绕过硬过滤 |

权重可以按风险和项目阶段调整，但调整规则必须确定且可测试。不能让语言模型直接输出无依据的总分。

每项推荐默认展示：

- 该项目为什么需要这项能力；
- 当前第一候选及其证据；
- 已存在的本地能力能否复用；
- 权限、数据范围和运行成本；
- 候选验证状态；
- 至少一个有实质差异的替代方案；
- 未推荐热门候选时的主要原因。

## 条件化示例

### 代码关系分析

CodeGraph 类能力不再仅由“大型 monorepo”开关决定。当模块多、跨语言调用复杂、变更影响难以用语言服务器回答或用户明确需要结构化关系查询时提升排名；小型单体仓库、已有可靠语言服务器且索引成本较高时降低排名。

### 跨会话记忆

长期维护、多人协作、决策密集或频繁跨会话时提升 Claude-Mem 类能力；短期探索、敏感数据无法持久化或项目已有规范化决策日志时降低排名。推荐必须说明存储位置、保留周期和项目隔离方式。

### 浏览器能力

已有 Playwright/Cypress 且目标是 CI 可重复测试时优先复用项目工具；需要交互式 DOM、网络、性能或页面状态调试时，才提高 Chrome DevTools MCP 类候选。两者可以互补，但不能把 MCP 自动等同于完整自动化测试体系。

### 设计与代码优化

已有经过验证的 Superpowers 或等价工作流时将其绑定到需求澄清、设计、计划、调试和验证阶段；代码优化候选则根据语言、性能目标、静态分析能力和现有质量工具单独选择。设计工作流与代码优化工具不得合并成一个模糊的“AI 编码能力”。

## 审批与状态机约束

初始化只有满足以下条件才能展示最终项目 ChangeSet 审批：

- 所有必要能力已经由本地提供者满足、选择了远程候选，或被用户明确延后/拒绝；
- 对值得搜索的重要缺口，用户已经明确批准或拒绝远程发现；
- 推荐中不存在被错误标记为用户决定的推断权限；
- 每个待安装候选具有精确身份、权限和文件变更；
- 发现批准、制品获取批准和安装批准保持独立。

如果上述条件未满足，流程停留在建议或发现阶段，不能用一个仅包含蓝图文件的 ChangeSet 掩盖未解决的开发能力闭环。

## 错误处理

- 权限回答未能映射时标记为 `unknown` 并重新询问，禁止默认拒绝。
- 旧 checkpoint 缺少新权限字段时执行显式迁移，不能从旧摘要反推用户决定。
- 部分远程来源失败时保留其他来源结果和逐来源诊断。
- 搜索无结果时保留静态方向，并允许用户补充产品名或扩大已批准来源。
- 候选全部被过滤时说明过滤类别，但不暴露恶意内容。
- 仓库变化、用户偏好变化、候选版本变化或权限变化会使旧推荐与旧安装批准失效并触发重算。

## 数据产物

项目初始化产物需要区分：

- `blueprint`：项目事实、目标、偏好、权限和来源；
- `capability-needs`：抽象能力需求、证据和强度；
- `recommendations`：本地与远程候选、评分、解释和状态；
- `decisions`：用户接受、拒绝、延后和锁定的决定；
- `discovery`：查询、来源、时间、诊断和缓存状态；
- `lock`：最终安装候选的不可变身份与摘要。

生成的 `AGENTS.md` 和项目能力管理 Skill 只引用已确认决策，不把未选择的静态线索写成可用能力。

## 测试与验收

至少覆盖以下场景：

1. 用户未回答网络问题时，报告 `unknown`，不得显示“禁用网络”。
2. 访谈线程被禁止联网时，不影响项目网络策略。
3. 用户未批准远程发现时显示 `not-requested` 和静态线索，并在 ChangeSet 前请求发现决定。
4. 空白全栈 Web 项目能得到设计、实现、测试、质量、连续性和恢复阶段的能力分析，而不是固定产品套餐。
5. 已安装 Superpowers、Chrome MCP 或其他提供者被本地盘点并参与能力竞争。
6. 同一项目在有无 Playwright、仓库规模变化和不同记忆偏好下产生不同推荐。
7. 普通但跨模块复杂的仓库可以推荐 CodeGraph 类候选；简单大型仓库也可以因收益不足而不推荐。
8. `quality.gates` 和 `repository.exploration` 不再产生无候选、无解释的裸缺口。
9. 用户提及未知或疑似拼写错误的产品名时，系统将其作为搜索线索并说明不确定性。
10. 未完成候选选择时无法进入最终安装 ChangeSet 审批。
11. 远程热度较高但权限过宽的候选不能超过低权限、高适配候选。
12. Doctor 能报告未知权限、未解决必要能力、过期发现证据和推荐/锁文件漂移。

## 与现有远程发现设计的关系

本设计复用《可信多来源远程能力发现设计》的来源、规范化、过滤、排序证据和审批边界，并在其上补齐三个上游能力：

1. 正式、可追溯的权限与偏好模型；
2. 证据驱动的抽象能力需求和动态提问；
3. 推荐完成前禁止进入安装 ChangeSet 的对话状态机。

远程发现仍是推荐系统的一个受控阶段，而不是推荐逻辑本身。
