# 可信多来源远程能力发现设计

**日期：** 2026-07-22

**状态：** 用户已批准设计，等待中文规格复核

**产品入口：** Codex 原生 Bootstrap Skill 与确定性的 `vibe` 能力工具

## 问题

当前初始化流程把 `--remote-discovery` 当成读取预生成
`.ai-project/remote-candidates.json` 快照的许可，并不会查询真实远程来源。快照不存在时，
命令返回空候选集合，外层 Codex 对话可能错误地报告“远程发现没有返回经过验证的候选”。

这混淆了多种本质不同的状态：

- 用户尚未批准远程发现；
- 没有配置可用的发现来源；
- 来源不可用、超时或受到限流；
- 搜索成功但没有结果；
- 搜索得到结果，但全部被安全策略过滤；
- 已有可用候选，但用户尚未批准安装。

产品必须只在确认本地能力缺口并获得明确发现授权后执行真实远程发现。发现授权不得隐含安装授权。

## 目标

1. 在确认本地能力缺口并获得用户授权后查询可信远程来源。
2. 支持官方 MCP Registry、skills.sh、GitHub 和已配置的组织级 Registry。
3. 保持本地优先；未执行网络发现时仍展示静态候选方向。
4. 以确定性方式规范化、去重、验证、过滤、排序并解释候选。
5. 将社区热度作为排序信号，但不允许热度绕过安全约束。
6. 每个能力缺口默认最多展示三个候选，其他合格候选按需展开。
7. 保留项目级安装、不可变身份、权限审阅、事务写入、回滚、审计和 Doctor 验证。

## 非目标

- 自动安装任何远程能力。
- 把 GitHub Star、skills.sh 使用量或被 MCP Registry 收录视为安全证明。
- 在不存在本地缺口或用户未授权时搜索网络。
- 在盘点或搜索阶段执行候选能力代码。
- 构建通用互联网搜索引擎。
- 替代 Codex 对普通开发任务进行原生 Skill 选择。

## 用户体验

Bootstrap Skill 使用以下流程：

```text
分析仓库和本地 Codex 环境
→ 推导抽象能力需求
→ 解析已验证的本地提供者
→ 确认真正的能力缺口
→ 展示静态候选方向，并说明尚未验证
→ 询问是否允许访问可信远程来源
→ 搜索已批准的来源
→ 规范化、去重、验证元数据并执行策略过滤
→ 为每个缺口展示前三名候选
→ 用户查看、替换、拒绝、延后或选择候选
→ 展示准确的安装权限与文件 Diff
→ 单独获得安装批准
→ 项目级安装、验证、锁定、审计并运行 Doctor
```

远程发现前，对话可以表述为：

> 当前仍缺少浏览器验证能力。本地候选方向包括 Playwright 和 Chrome DevTools MCP，
> 但尚未查询或验证任何远程来源。是否允许我从已批准来源搜索项目级候选？

只有至少一个来源实际成功完成查询后，系统才可以表述“搜索没有返回经过验证的候选”。

## 架构

### 发现服务

新增确定性的发现服务，输入包括：

- 抽象能力缺口；
- 仓库快照和 Blueprint；
- 用户批准的来源配置；
- 组织策略；
- 离线和缓存策略；
- 每个来源的超时与结果上限。

服务调用来源 Adapter，将结果规范化为 `RemoteCandidate`，附加来源证据，合并跨来源重复项，
执行硬性策略过滤，并返回结构化 `DiscoveryReport`。

发现服务不安装制品，也不执行任何被发现的代码。

### 来源 Adapter

所有来源 Adapter 使用现有 `RegistrySource` 风格的搜索和获取边界，同时返回来源诊断信息。

首批 Adapter：

1. **MCP Registry**：复用现有 Registry 客户端，读取官方 MCP Server 元数据。
2. **skills.sh**：读取 Skill 目录元数据、采用量、规范源仓库、版本或 Commit 身份和安装形态。
3. **GitHub**：发现可能提供 Skill、Plugin、MCP Server 或 CLI Tool 的仓库，并读取维护和社区信号。
   存在认证时使用认证只读访问，否则使用匿名只读访问。
4. **组织 Registry**：通过统一来源接口读取明确配置的内部或批准目录。

来源失败彼此隔离。单个来源失败不得清除其他来源已经成功返回的结果。

### 快照与缓存边界

`.ai-project/remote-candidates.json` 改为发现结果缓存和可复现快照，而不是发现实现本身。它记录：

- 查询的能力和来源；
- 获取时间和缓存状态；
- 规范化候选元数据；
- 可用的不可变来源身份；
- 排序证据；
- 来源诊断信息；
- 不包含 Secret 的策略过滤摘要。

离线初始化可以复用未过期快照，但必须明确标记结果来自缓存。仅仅缺少快照只能表示
`not-requested` 或 `source-unavailable`，不能表示 `no-results`。

## 发现状态

每个能力缺口必须具有一个明确的发现状态：

- `not-requested`：用户尚未批准远程发现；
- `source-unavailable`：没有配置获批来源，或来源不可访问；
- `search-failed`：所有已尝试来源均失败；
- `no-results`：至少一个来源成功完成搜索，但没有匹配记录；
- `all-filtered`：存在匹配记录，但全部未通过硬性策略或安全检查；
- `candidates-found`：至少存在一个可以展示的合格候选；
- `installation-deferred`：用户决定保留缺口，稍后处理；
- `installation-rejected`：用户对当前项目明确拒绝现有候选。

报告同时保存逐来源结果，使部分成功、限流和认证失败保持可见。

## 候选规范化与去重

不同来源的候选只有满足以下不可变或高可信身份条件时才能合并，优先级依次为：

1. 规范仓库地址与 Commit 或 Release 摘要完全一致；
2. 包标识、版本和摘要完全一致；
3. 来源提供明确的跨目录规范引用。

名称相同不能作为合并依据。摘要或发布者冲突的候选保持独立，并生成供应链冲突提示。

合并后的候选保留全部来源引用，使用最强的已验证证据，但不得丢弃较弱或冲突证据。

## 硬性过滤

满足以下任一条件的候选不得进入默认推荐列表：

- 无法确认来源或制品位置；
- 安装前无法固定版本、Commit 或验证后的内容摘要；
- 不同来源报告的仓库、摘要或发布者身份发生冲突；
- 仓库已经归档，且没有获批的维护中继任版本；
- 静态扫描发现提示注入或绕过安全控制的内容；
- 请求无关 Secret、未经授权的项目外写入或 L4 权限；
- 声明行为与实际观察到的包行为存在实质差异；
- 权限超过 Blueprint 或组织策略上限；
- 发布者、来源或能力被组织策略阻止；
- 广告中的能力无法由检查到的元数据或内容证明。

最终制品摘要尚不可用的候选可以参与发现，但必须标记为 `metadata-only`。只有获取并完成摘要、
发布者、权限和静态扫描验证后，候选才可以进入安装批准阶段。

## 排序

通过硬性过滤的候选分别计算五项可解释分数：

| 维度 | 权重 | 证据 |
| --- | ---: | --- |
| 适配度 | 35 | 能力匹配、项目事实、平台和运行时兼容性 |
| 可信度 | 25 | 来源等级、发布者验证、不可变身份、跨来源一致性 |
| 风险 | 20 | 权限等级、静态扫描、外部写入、凭据和可执行内容 |
| 维护度 | 10 | 最近 Release/Commit、归档状态、Issue 响应和贡献者 |
| 热度 | 10 | GitHub Star/Fork/依赖量及 skills.sh/MCP 采用信号 |

风险只在通过硬性过滤后转换为安全贡献分，残余风险越低排名越高。原始社区指标使用对数归一化并设置
上限，避免超大型项目仅凭热度压过更适合、更可信的候选。

热度不能：

- 绕过任何硬性过滤；
- 补偿无法验证的可执行内容；
- 使候选超过项目权限上限；
- 将 `metadata-only` 候选变成可安装候选。

GitHub 候选只有满足至少一项发现可信条件时，才进入默认前三名：

- 来自官方组织或已验证发布者；
- 同时被获批 Registry 收录；
- 获得组织配置批准；
- 达到配置要求的热度和维护度证据。

低热度但适配度高的候选只要通过全部安全过滤，仍可作为次级结果展示。当合格候选不足三个时，
它可以进入默认列表，但必须明确说明其社区证据较弱。

## 审批边界

系统使用两次独立批准：

1. **发现批准**：授权针对当前项目初始化或当前能力缺口，以只读方式访问明确列出的网络来源。
2. **安装批准**：授权一个准确候选、不可变版本、权限集合和项目级 ChangeSet。

发现批准不自动授权下载可执行制品进行预检。用户请求查看候选详情时可以进行只读制品获取，但必须提前说明。
候选版本、摘要、发布者、权限或文件 Diff 发生变化时，原安装批准立即失效。

## 错误处理与降级

- 来源超时、认证失败或限流会生成来源诊断，但不丢弃其他成功来源的候选。
- 所有来源失败时，状态为 `search-failed`，而不是 `no-results`。
- 没有获批来源时，状态为 `source-unavailable`。
- 一个来源成功返回零结果、另一个来源失败时，整体状态为 `no-results`，并附带部分失败警告。
- 所有候选均被策略过滤时，状态为 `all-filtered`，只展示过滤类别摘要，不暴露恶意内容或凭据。
- 所有非成功状态仍保留本地静态候选方向，并明确标注为未验证线索。
- 用户明确延后时，初始化可以保留未解决缺口继续进行；必要能力缺口必须写入生成配置和 Doctor 结果。

## Bootstrap Skill 行为

Bootstrap Skill 必须：

- 区分静态候选线索和经过远程验证的候选；
- 展示已确认缺口后再请求发现批准；
- 明确说明请求访问的来源和网络范围；
- 调用确定性发现接口，不能自行推断搜索结果；
- 展示来源状态、排序解释、权限和替代方案；
- 不得把缓存缺失等同于成功搜索为空；
- 安装或权限变化前获得单独批准；
- 保持当前 Codex 对话，不启动嵌套 Codex 任务。

## 测试与验收

单元测试必须覆盖：

- MCP Registry、skills.sh、GitHub 和组织 Registry 的结果规范化；
- 各来源的分页、限流、缓存和畸形响应；
- 跨来源去重和摘要冲突；
- 硬性过滤和组织策略；
- 热度的对数归一化与贡献上限；
- 相同分数下的稳定确定性排序；
- 所有发现状态和部分来源结果。

端到端测试必须覆盖：

- 缺少本地能力、用户未批准发现，并验证网络请求数为零；
- 用户批准发现，真实 Adapter Fixture 响应产生已排序候选；
- GitHub 与 skills.sh 重复候选被正确合并；
- 高热度但高风险的候选被硬性过滤；
- 热度较低但明显更适配的候选排名第一；
- 单个来源限流时，其他来源结果仍然可用；
- 所有来源不可用时仍保留静态候选线索；
- 发现批准与安装批准相互独立；
- 候选在审阅和安装之间发生变化时，原批准失效；
- 项目级安装、锁定、Doctor、卸载和回滚闭环。

发布验收必须包含一次真实 Codex 人工见证初始化，证明对话会在发现前询问用户、展示多来源排序候选、
在安装前再次询问，并准确区分来源失败与搜索为空。

## 迁移

现有 Fixture 快照继续作为缓存发现输入受到支持。当前把快照加载描述为远程发现的测试必须重命名或扩展，
避免把离线夹具覆盖误称为真实来源覆盖。

应复用现有 MCP `RegistryClient`、评分、来源验证、静态扫描、安装、审计和回滚组件，而不是重新实现。

首轮实现不删除 `--remote-discovery`，而是将它重新定义为允许运行已配置的发现服务。新增明确的离线或
仅缓存选项，以保持 Fixture 和 CI 的确定性行为。
